# SmartHealth Hub — Database Operations Guide

Reference for running Alembic migrations and the development seed script.

## Prerequisites

1. **PostgreSQL** running and reachable (Docker Compose recommended):
   ```bash
   docker-compose -f infra/docker-compose.yml up db -d
   ```

2. **Backend `.env` file** configured at `backend/.env`:
   ```
   DATABASE_URL=postgresql+asyncpg://shh_admin:changeme@localhost:5432/smarthealthhub
   ```

3. **Python dependencies** installed inside the backend virtual environment:
   ```bash
   cd backend
   pip install -e ".[dev]"
   # or
   pip install -r requirements.txt
   ```

All Alembic commands must be run from the **`backend/`** directory so that
`alembic.ini` and the `app` package are discoverable.

---

## Applying Migrations

### Apply all pending migrations (upgrade to latest)

```bash
cd backend
alembic upgrade head
```

### Apply one migration at a time

```bash
cd backend
alembic upgrade +1
```

### Check current revision

```bash
cd backend
alembic current
```

### View migration history

```bash
cd backend
alembic history --verbose
```

---

## Creating New Migrations

After adding or changing SQLAlchemy model definitions in `backend/app/models/`,
generate an autogenerated migration:

```bash
cd backend
alembic revision --autogenerate -m "add_column_to_patients"
```

Review the generated file in `backend/alembic/versions/` before applying it —
autogenerate cannot detect all changes (e.g., CHECK constraint changes,
server-default modifications, or custom PostgreSQL types).

---

## Rolling Back

### Roll back one migration

```bash
cd backend
alembic downgrade -1
```

### Roll back to a specific revision

```bash
cd backend
alembic downgrade 0001
```

### Roll back everything (base state — no tables)

```bash
cd backend
alembic downgrade base
```

## Seeding Development Data

The seed script populates roles, users, and 8 sample Filipino patients with
medical history, immunization, and appointment records.

```bash
# From the backend/ directory:
cd backend && python ../scripts/seed_db.py

# Or from the repo root:
python scripts/seed_db.py
```

Default credentials created by the seed (local development only):

| Role  | Email                           | Password  |
|-------|---------------------------------|-----------|
| admin | admin@smarthealthhub.local      | admin123! |
| bhw   | bhw@smarthealthhub.local        | bhw123!   |

**Change these credentials before deploying to any shared or production environment.**

## Full Local Dev Reset

Destroys the database volume, recreates it from scratch, runs all migrations,
and re-seeds development data:

```bash
docker-compose -f infra/docker-compose.yml down -v
docker-compose -f infra/docker-compose.yml up db -d
cd backend
alembic upgrade head
python ../scripts/seed_db.py
```

## Schema Overview

12 tables, created in FK dependency order:

| # | Table              | Purpose                                                  |
|---|--------------------|----------------------------------------------------------|
| 1 | `roles`            | RBAC role definitions with JSONB permissions             |
| 2 | `users`            | Staff accounts (Admin, BHW, Physician, Admin Staff)      |
| 3 | `patients`         | Core patient demographic records                         |
| 4 | `medical_history`  | Chronic conditions; `notes` encrypted AES-256-GCM        |
| 5 | `immunizations`    | Vaccine dose records; drives SMS reminder scheduler      |
| 6 | `appointments`     | Scheduled consultations; drives SMS appointment reminders|
| 7 | `visits`           | Clinical visit notes; `diagnosis` + `treatment_notes` encrypted |
| 8 | `health_cards`     | NFC/QR card metadata (pointer only — no PHI in payload)  |
| 9 | `card_verifications` | Audit log of every NFC tap / QR scan                  |
| 10| `mfa_otp`          | Argon2-hashed OTP codes for MFA login and password reset |
| 11| `sms_logs`         | SMS dispatch records; updated by Semaphore webhook       |
| 12| `audit_logs`       | Append-only audit trail for all PHI access/mutations     |

Full column definitions and ER diagram: `docs/SmartHealth_Hub_System_Development_Plan.md` Section 4.

## Application-Layer Encryption

Three columns store ciphertext (AES-256-GCM) encrypted before INSERT:

- `medical_history.notes`
- `visits.diagnosis`
- `visits.treatment_notes`

Encryption/decryption is handled by `backend/app/utils/encryption.py`
(Phase 1 deliverable).  The `ENCRYPTION_KEY` setting in `.env` holds the
Base64-encoded 32-byte key.  Never commit a real key to source control.

---

## Audit Log Policy

The `audit_logs` table is append-only:
- The application service role has no `UPDATE` or `DELETE` privilege on this table.
- Every CREATE/UPDATE/DELETE on patient records, medical history, and health
  cards writes one row to `audit_logs`.
- Every PHI read (medical history notes, visit diagnosis) writes an
  `action = "VIEW_PHI"` row.
